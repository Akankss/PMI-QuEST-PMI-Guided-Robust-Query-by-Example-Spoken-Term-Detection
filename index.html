Step 03 of 07 — Select PMI bigrams

PMI measures how much more adjacent token pairs co-occur than expected under independence. Only pairs exceeding threshold τ = 0.5 enter the index.

PMI formula
PMI(a,b) = log   p(a,b) / (p(a)·p(b))

// p(a,b) = adjacent-pair corpus frequency
// PMI > 0 : appear together more than chance → kept if > τ
// PMI ≤ 0 : accidental adjacency → discarded

𝒷_τ = { (a,b) ∈ 𝒱² : PMI(a,b) > τ }

Step 04 of 07 — Build the PMI-TF-IDF vector

Concatenate unigram TF-IDF weights with α-scaled bigram weights, then L2-normalise. The resulting vector encodes both what tokens appear (unigrams) and in what order (bigrams).

Unigram part u_q — TF-IDF weights
w_uni(t, s) = (n_t(x) / |s(x)|) · log(N / (1+d_t))

token 271 (/w/) : 1/5 × 2.3 = 0.46
token 089 (/ʌ/) : 1/5 × 3.4 = 0.68 ← high IDF, rare
token 314 (/n/) : 1/5 × 2.8 = 0.56
token 452 (/d/) : 1/5 × 2.1 = 0.42
token 177 (/ɚ/) : 1/5 × 3.1 = 0.62
Bigram part b_q — α = 1.0
w_bi((a,b), s) = (n_ab(x)/(|s|-1)) · log(N/(1+d_ab))

(089→314) : 1/4 × 3.4 = 0.85 × α = 0.85
(314→452) : 1/4 × 2.8 = 0.70 × α = 0.70
(452→177) : 1/4 × 3.1 = 0.78 × α = 0.78

PMI Bigram Discovery
Pointwise Mutual Information measures how much more a token pair co-occurs than expected under independence. Only statistically surprising adjacent pairs enter the index.
The PMI Formula
PMI(a, b) = log   p(a, b)
                   p(a) · p(b)

// p(a,b) = adjacent-pair corpus frequency
// PMI > 0 → co-occur more than chance → kept if > τ
// PMI ≤ 0 → accidental adjacency → discarded

𝒷_τ = { (a,b) ∈ 𝒱² : PMI(a,b) > τ }
Why not frequency (BPE)?

BPE greedily merges the most frequent adjacent pairs. In a speech corpus these are widespread phonotactic transitions shared across many words they carry near-zero IDF and contribute noise rather than discriminative signal.

schwa → /t/ freq-high, IDF≈0
/ə/ → /n/ common across 1000s of words
Why PMI works

PMI normalises by marginal frequencies. A bigram passes only when both tokens appear together significantly more than their individual rates predict, regardless of absolute frequency.

/k/ → /æt/ PMI=2.1, word-specific
/st/ → /ɹ/ PMI=1.7, phoneme cluster
Vocabulary compactness (K=1024, τ=0.5)

PMI filtering reduces the candidate bigram space by three orders of magnitude:

All possible bigrams (K²)1,048,576
PMI-selected 𝒷_τ< 1,400

The bigram bar is genuinely tiny that's the point. Only the rarest, most informative token transitions survive.

Rare-event bias & self-correction

Raw PMI can favour very rare bigrams. In PMI-QuEST this is self-correcting: constituents of rare bigrams are also rare, so their IDF weights are large. SW alignment is not misled. τ=0.5 yields stable, degenerate-free bigram sets across all tokeniser configs; normalised PMI (NPMI) is a viable drop-in for richer corpora.

Why BPE underperforms
BPE's failure mode

BPE optimises for compression, not discrimination. It absorbs the most frequent token pairs common phonotactic transitions (e.g. schwa + coronal stop) shared across thousands of words into single merged tokens. These carry near-zero IDF and contribute noise rather than discriminative signal.

PMI's mechanism

PMI identifies statistically surprising co-occurrences. These tend to correspond to word-specific or morpheme-specific phoneme clusters — exactly the signal needed for term detection. A bigram passes only when its joint probability exceeds the product of its marginals, irrespective of absolute frequency.

Why it generalises

PMI bigram selection is entirely data-driven and language-agnostic: statistics are estimated independently from each target-language corpus, requiring no cross-lingual transfer of the bigram vocabulary. XLS-R-300M (pretrained on 128 languages) provides phonemic coverage; PMI then captures language-specific token collocations that distinguish word identities. Punjabi shows the smallest gain (+0.056), attributed to its close phonological similarity to Hindi, which increases within-corpus ambiguity.

PMI-QuEST Algorithm
Two-phase retrieve-then-rerank pipeline. The offline phase runs once per corpus. The online phase runs per query in O(log N + C·m·n̄) time.
ALGORITHM PMI-QuEST
INPUT: Corpus 𝒟, encoder f_ℓ, centroids {μ_c}, threshold τ, weight α, C, SW params (m₊, m₋, g)
OUTPUT: Ranked list for query x_q

// ── OFFLINE: executed once per corpus ──────────────────────
FOR each x_d ∈ 𝒟:
s_d ← TOKENISE(x_d, f_ℓ, {μ_c})

Estimate p(t) and p(a,b) from corpus adjacent-pair counts
𝒷_τ ← { (a,b) : PMI(a,b) > τ } // filter bigrams
Compute IDF(t) for all t ∈ 𝒱 ; IDF(a,b) for all (a,b) ∈ 𝒷_τ

FOR each x_d ∈ 𝒟:
v_d ← [ u_d ‖ α·b_d ] / ‖·‖₂ // PMI-TF-IDF vector

Build HNSW index ℋ over {v_d} // M=16, ef_construction=150

// ── ONLINE: executed per query ─────────────────────────────
s_q ← TOKENISE(x_q, f_ℓ, {μ_c})
v_q ← [ u_q ‖ α·b_q ] / ‖·‖₂
𝒞 ← ℋ.SEARCH(v_q, C) // ef_search=200, C=200

FOR each d ∈ 𝒞:
H(i,j) = max(0, H(i-1,j-1)+σ(t_i^q, t_j^d), H(i-1,j)+g, H(i,j-1)+g)
SW(q,d) ← max_{i,j} H(i,j) / m

RETURN 𝒞 sorted by SW(q,d) descending
Complexity
Offline: O(N·L) tokenisation + O(N) vector build
         + O(N log N) HNSW construction

Online: O(log N) HNSW search
         + O(C·m·n̄) SW reranking

// vs DTW: O(N·m·L) — no pre-filter
